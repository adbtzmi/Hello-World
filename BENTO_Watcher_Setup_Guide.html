🍱 BENTO Tester Watcher Setup Guide v1.3

How to deploy and configure the BENTO watcher script on a new tester machine for automated TP compilation

Micron Technology | SSD Firmware Test Engineering

📖 Overview

🔍 What is the BENTO Tester Watcher?

The BENTO Tester Watcher is a Python script that runs continuously on a tester machine (Advantest). It watches a shared network folder for incoming TP packages (ZIP files) generated by BENTO on the local PC, compiles them using make release (or make release_supermicro for CNFG), and places the resulting TGZ file back in the shared release folder.


This guide covers:

  • Prerequisites — what must exist on the tester before setup
  • Verifying the watcher files on the shared folder
  • Copying the TP repository to the local PC and tester machine
  • Verifying the shared folder connection
  • Running the watcher manually to test
  • Configuring Windows Task Scheduler for automatic startup
  • Adding the tester to the BENTO GUI
  • Troubleshooting

🏗️ Architecture

Understanding the data flow helps diagnose issues:


💻 Local PC
Runs BENTO GUI (main.py). Creates ZIP and drops it in RAW_ZIP.
📁 Shared Folder
P:\temp\BENTO\ — visible from both local PC and all testers.
🖥️ Tester Machine
Runs watcher_main.py. Picks up ZIP, builds, copies TGZ to RELEASE_TGZ.

FolderPathPurpose
RAW_ZIPP:\temp\BENTO\RAW_ZIP\BENTO drops ZIPs here
RELEASE_TGZP:\temp\BENTO\RELEASE_TGZ\Tester places compiled TGZ here

📄 Files Reference

FileDescription
watcher_main.pyEntry point. Run with: python watcher_main.py --env ABIT (or SFN2 / CNFG)
watcher_config.pyAll paths and settings. Edit this for path changes.
watcher_builder.pyRuns make release via Cygwin bash. Handles memory cleanup.
watcher_copier.pyBinary copy of TGZ to shared release folder.
watcher_lock.pyLock file and build status logic.

🏷️ ENV Tokens

Each tester is identified by both its hostname AND its ENV token. The watcher only picks up ZIPs whose filename contains BOTH this tester's hostname AND its ENV token. This means two testers sharing the same ENV (e.g. two ABIT testers) will never claim each other's ZIPs.

SFN2MPT3HVM-0156make release
CNFGCTOWTST-0031make release_supermicro

✅ Prerequisites

Before running any steps, confirm all of the following are true on the tester machine:

Checklist

  • Cygwin is installed (typically C:\cygwin64\)
  • make, gcc, and bash are available in Cygwin (check: C:\cygwin64\bin\make.exe exists)
  • The TP repo exists at C:\BENTO\adv_ibir_master on the tester (or your team's configured path)
  • The shared folder P:\temp\BENTO is accessible from the tester (P: drive mapped)
  • P:\temp\BENTO\RAW_ZIP\ and P:\temp\BENTO\RELEASE_TGZ\ folders exist
  • Python is installed and accessible on Windows (run: python --version in Command Prompt)
  • You have VNC or remote desktop access to the tester
  • You have Administrator rights (needed for Task Scheduler)
ℹ If P: drive is not mapped on the tester, map it manually: open File Explorer → This PC → Map network drive → enter \\server\temp\BENTO as the target.

📦 Installation

📁 Step 1 · Locate the Watcher Files on the Shared Folder

All watcher files are already available on the shared network folder. You do NOT need to copy or transfer any files manually — every tester reads from the same location.


From the tester machine, verify you can see these files at:

P:\temp\BENTO\watcher\

Expected files:

  • watcher_main.py — entry point, run this to start the watcher
  • watcher_config.py — all paths and settings
  • watcher_builder.py — handles build execution
  • watcher_copier.py — handles TGZ copy to release folder
  • watcher_lock.py — manages lock and status files
ℹ If the files are not visible at P:\temp\BENTO\watcher\, check that the P: drive is mapped correctly on this tester. See the Prerequisites section for drive mapping instructions.
💡 There is no installation required. The watcher runs directly from the shared folder. All testers always use the latest version automatically.

📦 Step 2 · Copy the TP Repository

The shared folder contains a pre-configured copy of the TP repository at P:\temp\BENTO\adv_ibir_master. This copy has been specifically modified for compilation purposes (the Makefile tar compression was changed to use a pipe-based approach to prevent OOM kills on tester machines). You must use this copy — not the standard Bitbucket clone — for BENTO compilation.


Part A — Copy to the Tester Machine

From the tester machine, copy the TP repo from the shared folder to the local C:\BENTO directory:

xcopy /E /I /H P:\temp\BENTO\adv_ibir_master C:\BENTO\adv_ibir_master

After copying, the tester repo dir will be C:\BENTO\adv_ibir_master. This is the path you will enter when registering the tester in Step 7.


Part B — Copy to the Local PC

From your local PC, copy the same repo to a local folder of your choice (e.g. C:\Users\YourName\Repos\adv_ibir_master):

xcopy /E /I /H P:\temp\BENTO\adv_ibir_master C:\Users\YourName\Repos\adv_ibir_master

Then update the Repo Path in the BENTO GUI — go to the Implementation tab → AI Plan Generator tab → set the Local Repo Path field to the folder you just copied.

⚠ IMPORTANT — Two Separate Repo Paths: The copy from the shared folder is for BENTO compilation only. If you want to generate an Implementation Plan using the AI, you must switch back to your original Bitbucket-cloned repo path in the Implementation tab inside the AI Plan Generator tab. The shared folder copy uses a modified Makefile (different tar compression) optimised for tester compilation — it should not be used for code review or AI analysis.

🔧 Step 3 · Verify Cygwin and make are Available

The BENTO watcher uses Cygwin bash to run make release. Here is why both Cygwin and Windows Command Prompt are involved:


QuestionAnswer
Why Cygwin bash?The tools used to build TP packages (make, gcc, ld) are Cygwin tools. They live in C:/cygwin64/bin/ and only work inside a Cygwin environment. Windows Command Prompt cannot find or run them.
Why not just use Cygwin?Cygwin bash itself consumes significant RAM on top of the already heavy make release build. On tester machines with limited memory, this can cause the OS to kill the process (OOM kill) before the TGZ copy completes.
What BENTO doesBENTO calls Cygwin bash -l (login shell) to load the full Cygwin environment and PATH, runs make release inside it, then the bash process exits. The subsequent TGZ copy runs as a lightweight Python operation — much lower RAM, very unlikely to be OOM killed.

To verify Cygwin and make are correctly installed, open a Windows Command Prompt on the tester and run:

REM Check Cygwin bash exists:
dir C:\cygwin64\bin\bash.exe

REM Check make is available inside Cygwin:
C:\cygwin64\bin\bash.exe -lc "which make"
REM Expected: /usr/bin/make

C:\cygwin64\bin\bash.exe -lc "make --version"
REM Expected: GNU Make 4.x or similar

If make is not found, install it via Cygwin:

1
Run C:\cygwin64\setup.exe
2
Search for make in the package list and select it for installation
3
Also ensure these are installed: gcc-core, gcc-g++, binutils
ℹ If Cygwin is installed in a non-standard path (e.g. C:\cygwin\ instead of C:\cygwin64\), update find_cygwin_bash() in watcher_builder.py to add your path.

📋 Step 4 · Verify the Tester Registry

Tester settings (hostname, repo path, and build command) are managed through the BENTO GUI — you do NOT need to edit watcher_config.py directly. The watcher loads its configuration automatically from P:\temp\BENTO\bento_testers.json at startup.


SettingValue
RAW_ZIP_FOLDERP:\temp\BENTO\RAW_ZIP
RELEASE_TGZ_FOLDERP:\temp\BENTO\RELEASE_TGZ
repo_dir (ABIT)C:\BENTO\adv_ibir_master
build_cmd (ABIT)make release
build_cmd (CNFG)make release_supermicro
POLL_INTERVAL60 (seconds)
LOG_DIRC:\BENTO\logs
💡 To verify the registry, open P:\temp\BENTO\bento_testers.json in Notepad and confirm this tester's entry is present with the correct hostname, repo_dir, and build_cmd. If it is missing or incorrect, use the + Add Tester dialog in the BENTO GUI (Step 7) to register the tester — the GUI writes the file automatically.

▶️ Running the Watcher

🧪 Step 5 · Run Manually to Test

Before setting up Task Scheduler, always run the watcher manually first to confirm it works.


Open a Windows Command Prompt on the tester and run:

cd P:\temp\BENTO\watcher

# For ABIT tester:
python watcher_main.py --env ABIT

# For SFN2 tester:
python watcher_main.py --env SFN2

# For CNFG tester:
python watcher_main.py --env CNFG

Expected startup output:

2026-03-12 08:17:22 | INFO     | ========================
2026-03-12 08:17:22 | INFO     | BENTO Tester Watcher  |  Environment: ABIT
2026-03-12 08:17:22 | INFO     | Hostname   : IBIR-0383
2026-03-12 08:17:22 | INFO     | RAW_ZIP    : P:\temp\BENTO\RAW_ZIP
2026-03-12 08:17:22 | INFO     | ========================
2026-03-12 08:17:22 | INFO     | Startup lock cleanup done. Removed: 0
2026-03-12 08:17:22 | INFO     | Watching P:\temp\BENTO\RAW_ZIP every 60s ...

Now trigger a compile from BENTO on your local PC and watch the log output. If it completes successfully, proceed to Step 6.

ℹ Press Ctrl+C to stop the watcher after testing. It does NOT need to stay running during Task Scheduler setup.

⏰ Step 6 · Configure Windows Task Scheduler

Task Scheduler will start the watcher automatically whenever the tester reboots, running it in the background as an Administrator.


1
Open Task Scheduler — Start → search 'Task Scheduler' → open
2
Click 'Create Task...' (not 'Create Basic Task') in the right panel

📋 General Tab

  • Name: BENTO Watcher ABIT (use your ENV)
  • Description: BENTO tester watcher for TP package compilation
  • Select: 'Run whether user is logged on or not'
  • Check: 'Run with highest privileges'
  • Configure for: Windows 10 or your OS version

⏱️ Triggers Tab

  • Click New → Begin the task: 'At startup'
  • Check: Enabled
  • Optional: add a 60-second delay (allows network drives to map before watcher starts)

▶️ Actions Tab

  • Click New → Action: 'Start a program'
FieldValue
Program/scriptpython
Add argumentswatcher_main.py --env ABIT
Start in (required)P:\temp\BENTO\watcher

⚙️ Settings Tab

  • Check: 'Allow task to be run on demand'
  • If task is already running: 'Do not start a new instance'
  • Uncheck: 'Stop task if it runs longer than' (watcher runs indefinitely)

Click OK and enter your Windows credentials when prompted.

💡 To verify the task runs: right-click it in Task Scheduler → Run. Check C:\BENTO\logs\ for the watcher log file.

🖥️ Adding the Tester to BENTO GUI

➕ Step 7 · Register Tester in BENTO

Once the watcher is running on the tester, add it to the BENTO GUI so it appears in the dropdown:


1
Open BENTO on your local PC
2
Go to the Implementation tab
3
In the 'Compile TP Package' section, click '+ Add Tester'
4
Fill in all four fields (all required):
  • Tester Hostname: the machine name (e.g. IBIR-0999)
  • Environment: select ABIT, SFN2, or CNFG
  • Repo Path: full path to the TP repo on this tester (e.g. C:\BENTO\adv_ibir_master) — use the 📁 button to browse
  • Build Command: select make release or make release_supermicro (CNFG always uses make release_supermicro)
5
Click 'Add Tester' — the button is disabled until all three preflight checkboxes are ticked

The new tester will appear in the dropdown immediately. The registry is saved locally next to main.py and synced to P:\temp\BENTO\bento_testers.json on the shared folder so the watcher can read it.

ℹ Restart the watcher on the tester for it to pick up the new entry.

🔧 Troubleshooting

This section covers common issues, their likely causes, and how to fix them.

📦 Watcher Does Not Pick Up ZIP

❌ ZIP hostname / ENV mismatch
What to look for
The watcher is running but ignores the ZIP file in RAW_ZIP.
Likely cause
ZIP filename must contain BOTH the tester hostname AND the ENV token. Format: <JIRA>_<HOSTNAME>_<ENV>_<DATE>_<TIME>.zip — e.g. TSESSD-14270_IBIR-0383_ABIT_20260312_0832.zip is only claimed by the watcher running --env ABIT on host IBIR-0383. A different ABIT tester (e.g. IBIR-0999) will not pick it up.
How to fix
Verify the ZIP filename matches the tester's hostname and ENV. Re-trigger the compile from BENTO with the correct tester selected.
❌ Watcher not running
How to fix
Check Task Scheduler → BENTO Watcher task status. Logs are in C:\BENTO\logs\
❌ P: drive not mapped
Likely cause
The tester cannot see RAW_ZIP.
How to fix
Re-map the P: drive or check network connectivity. See Prerequisites.
❌ Pre-existing ZIP
Likely cause
If the watcher started AFTER the ZIP was dropped, it ignores it.
How to fix
Drop a new ZIP by re-triggering the compile from BENTO.

🔨 Build Fails (make release)

❌ make not found
Likely cause
Cygwin bash cannot find make.
How to fix
Run: C:\cygwin64\bin\bash.exe -lc 'which make' — if not found, install it via Cygwin setup. See Step 3.
❌ Wrong repo path
How to fix
Check repo_dir in watcher_config.py. Path must exist on the tester.
❌ Build error
How to fix
Check the build log in C:\BENTO\logs\build_<ZIP_NAME>.log for compiler errors.
❌ OOM / killed
Likely cause
The watcher sleeps before and after the build to give the OS time to reclaim RAM. If still failing, the tester may not have enough memory.
How to fix
Increase PRE_BUILD_SLEEP_SECONDS (pre-build) or POST_BUILD_SLEEP_SECONDS (post-build) in watcher_config.py.

📤 TGZ Not Appearing in RELEASE_TGZ

❌ Copy failed
How to fix
Check watcher log for 'Binary copy failed'. Verify write permission on P:\temp\BENTO\RELEASE_TGZ.
❌ Wrong folder
Likely cause
TGZ is in RELEASE_TGZ\TSESSD-14270_ABIT\ subfolder, not the root.
❌ Size mismatch
Likely cause
Watcher verifies copy size. If mismatch, it removes the partial file and logs an error.
How to fix
Check the watcher log for details and re-trigger the build.

📧 Contact / Escalation

If you are unable to resolve the issue after following this guide:

  • Check the bento_status sidecar file — it contains the last error message from the tester
  • Share the build log (C:\BENTO\logs\build_*.log) with your team lead
  • Provide the watcher log for the relevant session
